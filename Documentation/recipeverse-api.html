What is an API? Think of an API like a waiter in a restaurant. Your app (the customer) tells the waiter what it wants. The waiter goes to the kitchen (the database), gets the food (data), and brings it back. Every button tap in RecipeVerse calls an API endpoint behind the scenes.
Part 1 — Authentication
POST /auth/signup
Sign up with email & password
Creates a brand new user account. Sends a verification email. Returns an auth token the app stores for future requests.
🌐 Public
Required inputs (request body)
emailstringUser's email address. Must be unique.
passwordstringMin 8 characters. Stored as a hash, never plain text.
display_namestringTheir full name shown on the profile.
usernamestringTheir @handle. Must be unique, no spaces.
Example success response (201 Created)
{
  "success": true,
  "token": "eyJhbGciOiJIUzI1...",
  "user": {
    "id": "usr_a1b2c3",
    "email": "amara@example.com",
    "username": "amaracooks",
    "email_verified": false
  }
}
💡
The token is like a key card. The app stores it and sends it with every future request to prove the user is logged in. email_verified starts as false until they click the link in their email.
POST /auth/login
Log in with email & password
Checks the email and password. If correct, returns a fresh auth token. This is called every time a returning user opens the app and enters their credentials.
🌐 Public
Required inputs
emailstringThe email they signed up with.
passwordstringTheir password. Compared against the stored hash.
Example success response (200 OK)
{
  "success": true,
  "token": "eyJhbGciOiJIUzI1...",
  "user": {
    "id": "usr_a1b2c3",
    "username": "amaracooks",
    "avatar_url": "https://cdn.rv.com/u/a1.jpg"
  }
}
💡
If email or password is wrong, return a 401 error with message "Invalid email or password." Never say which one is wrong — that would help hackers guess accounts.
Part 2 — Profiles
POST /profiles
Create profile (auto-called at signup)
Creates the public profile for a newly signed-up user. In most cases this is called automatically by the system right after signup — users don't trigger this manually.
🔐 Auth required
Inputs
usernamestringTheir chosen @handle. Checked for availability.
display_namestringFull name shown on the profile page.
avatar_urlstringURL to their profile photo (from Google if OAuth).
biostringShort description, max 160 characters.
locationstringE.g. "Lagos, Nigeria".
Example success response (201 Created)
{
  "id": "pro_x9y8z7",
  "user_id": "usr_a1b2c3",
  "username": "amaracooks",
  "display_name": "Amara Osei",
  "follower_count": 0,
  "following_count": 0,
  "recipe_count": 0
}
💡
This is almost never called directly by the frontend. It fires automatically inside the signup flow. Think of it as the system setting up the user's public page immediately after they create an account.
PATCH /profiles/me
Update my profile
Lets a logged-in user edit their own public profile — change their bio, location, avatar, or display name. Only send the fields you want to change.
🔐 Auth required
Inputs (send only what changes)
display_namestringNew display name.
biostringUpdated bio text, max 160 chars.
avatar_urlstringURL of new profile photo (uploaded first via /media/upload).
locationstringUpdated location text.
website_urlstringLink to their external website or blog.
Example success response (200 OK)
{
  "success": true,
  "profile": {
    "username": "amaracooks",
    "display_name": "Amara Osei-Bonsu",
    "bio": "West African food lover 🍛",
    "location": "Lagos, Nigeria",
    "updated_at": "2026-03-17T10:23:00Z"
  }
}
💡
PATCH means "partial update" — you only send the fields you're changing. You don't need to re-send all fields. This is different from PUT which would require sending the entire profile each time.
Part 3 — Recipes
POST /recipes
Create a new recipe
Saves a new recipe to the database. Works for all three creation methods: typed, voice-converted (AI pre-fills the fields), and social import. The status field controls if it goes live immediately or saves as a draft.
🔐 Auth required
Inputs
titlestringRecipe name. E.g. "Spicy Jollof Rice".
ingredientsarrayList of {name, quantity, unit, order_index}.
stepsarrayList of {step_number, instruction, image_url?}.
statusstring"published" or "draft".
descriptionstringShort intro about the recipe.
cuisinestringE.g. "Nigerian", "Italian".
difficultystring"easy", "medium", or "hard".
cook_time_minsintegerTotal cook time in minutes.
servingsintegerNumber of servings.
cover_image_urlstringURL from /media/upload done first.
tagsarrayList of tag names: ["vegan", "quick"].
source_urlstringOriginal URL if imported from social media.
Example success response (201 Created)
{
  "success": true,
  "recipe": {
    "id": "rec_p9q8r7",
    "title": "Spicy Jollof Rice",
    "slug": "spicy-jollof-rice",
    "status": "published",
    "author": {
      "username": "amaracooks",
      "avatar_url": "https://cdn.rv.com/u/a1.jpg"
    },
    "published_at": "2026-03-17T10:30:00Z"
  }
}
💡
For voice uploads: the AI processing happens before this API is called. The AI returns structured data (title, ingredients, steps) which pre-fills the form. The user reviews it, then this endpoint is called to save it. This API doesn't know or care how the recipe was created.
PATCH /recipes/:id
Edit an existing recipe
Updates any fields of an existing recipe. Only the recipe's author can edit it. Also used to change status from "draft" to "published" when ready to go live.
🔐 Auth required
Inputs (send only changed fields)
titlestringUpdated title.
descriptionstringUpdated description.
ingredientsarrayFull updated ingredients list (replaces old list).
stepsarrayFull updated steps list (replaces old list).
statusstring"published", "draft", or "archived".
cover_image_urlstringNew cover photo URL.
tagsarrayReplaces existing tag list.
Example success response (200 OK)
{
  "success": true,
  "recipe": {
    "id": "rec_p9q8r7",
    "title": "Spicy Jollof Rice (Updated)",
    "status": "published",
    "updated_at": "2026-03-17T11:00:00Z"
  }
}
💡
The :id in the URL path is the recipe ID. So editing recipe "rec_p9q8r7" would call PATCH /recipes/rec_p9q8r7. The server checks that the logged-in user is the recipe's author before allowing any changes.
GET /recipes/feed
Get the home feed
Returns a paginated list of recipes from creators the logged-in user follows, sorted newest first. New users with no follows get trending recipes instead so the feed is never empty.
🔐 Auth required
Query parameters (in the URL)
pageintegerPage number for pagination. Default: 1.
limitintegerRecipes per page. Default: 20. Max: 50.
cursorstringID of the last recipe seen (for infinite scroll).
Example: GET /recipes/feed?limit=20&cursor=rec_abc123
Example success response (200 OK)
{
  "recipes": [
    {
      "id": "rec_p9q8r7",
      "title": "Spicy Jollof Rice",
      "cover_image_url": "https://cdn.rv.com/r/p9.jpg",
      "cook_time_mins": 45,
      "difficulty": "medium",
      "like_count": 142,
      "avg_rating": 4.8,
      "is_liked_by_me": false,
      "is_saved_by_me": true,
      "author": { "username": "amaracooks" }
    }
  ],
  "next_cursor": "rec_m3n4o5",
  "has_more": true
}
💡
is_liked_by_me and is_saved_by_me are added for the logged-in user so the app knows whether to show a filled or empty heart/bookmark icon without making extra API calls. next_cursor is used for "load more" — pass it back as the cursor param to get the next batch.
GET /recipes/:id
Get full recipe details
Returns everything about one recipe — all ingredients, all steps, all media, the author's profile, ratings, and recent comments. This powers the recipe detail page.
🌐 Public
Inputs
:idURL paramThe recipe ID or slug in the URL path.
Example:
GET /recipes/rec_p9q8r7
GET /recipes/spicy-jollof-rice
Example success response (200 OK)
{
  "id": "rec_p9q8r7",
  "title": "Spicy Jollof Rice",
  "description": "A family favourite...",
  "cook_time_mins": 45,
  "servings": 4,
  "ingredients": [
    { "name": "long grain rice", "quantity": "2", "unit": "cups" }
  ],
  "steps": [
    { "step_number": 1, "instruction": "Blend tomatoes..." }
  ],
  "author": { "username": "amaracooks", "is_verified": false },
  "avg_rating": 4.8,
  "like_count": 142,
  "is_liked_by_me": false,
  "tags": ["nigerian", "rice"]
}
💡
This is the most important and most-called API in the whole app. Every time someone opens a recipe page, this fires. Keep it fast — cache it on a CDN for logged-out users. Also increments view_count in the background.
Part 4 — Social interactions
POST /follows/:userId
Follow / unfollow a user
POST to follow. DELETE to unfollow. When you follow someone, their recipes start appearing in your home feed. The followed user's follower_count goes up by 1 and they get a notification.
🔐 Auth required
Inputs
:userIdURL paramThe ID of the user to follow or unfollow.
Follow: POST /follows/usr_marco123
Unfollow: DELETE /follows/usr_marco123
Example success response (200 OK)
// After following:
{
  "success": true,
  "action": "followed",
  "following_id": "usr_marco123",
  "their_follower_count": 1043
}

// After unfollowing:
{
  "success": true,
  "action": "unfollowed"
}
💡
After a follow, two things happen in the background: (1) the followed user's follower_count in the profiles table goes up by 1, and (2) a notification row is created so they get an alert. Both happen server-side — the frontend just calls this one endpoint.
POST /likes/:recipeId
Like / unlike a recipe
POST to like a recipe. DELETE to unlike. Tapping the heart icon calls POST. Tapping it again calls DELETE. The like_count on the recipe updates instantly in the UI (optimistic update).
🔐 Auth required
Inputs
:recipeIdURL paramThe ID of the recipe being liked or unliked.
Like: POST /likes/rec_p9q8r7
Unlike: DELETE /likes/rec_p9q8r7
Example success response (200 OK)
{
  "success": true,
  "action": "liked",
  "recipe_id": "rec_p9q8r7",
  "new_like_count": 143
}
💡
Optimistic update means the frontend shows the heart as filled immediately when tapped — without waiting for the server response. If the API call fails, it reverts. This makes the app feel instant and responsive even on slow connections.
POST /recipes/:recipeId/comments
Post a comment (or reply)
Adds a comment to a recipe. If parent_comment_id is included, it's a reply to that comment. Returns the new comment so the UI can display it instantly without refetching all comments.
🔐 Auth required
Inputs
bodystringThe comment text. Min 1 character, max 1000.
parent_comment_idstringInclude this to make it a reply. Leave out for a top-level comment.
Example success response (201 Created)
{
  "success": true,
  "comment": {
    "id": "cmt_j1k2l3",
    "body": "Made this last night — amazing!",
    "parent_comment_id": null,
    "author": {
      "username": "jiyeon_eats",
      "avatar_url": "https://cdn.rv.com/u/jy.jpg"
    },
    "created_at": "2026-03-17T14:05:00Z"
  }
}
💡
After a comment is posted, a notification is created for the recipe's author so they know someone commented. If it's a reply, the notification goes to the parent comment's author instead.
POST /saved/:recipeId
Save / unsave a recipe to a collection
POST to save a recipe to a named collection (like a bookmark). DELETE to remove it. If collection_name is not provided, it saves to the default "All Saved" collection.
🔐 Auth required
Inputs
:recipeIdURL paramRecipe to save or unsave.
collection_namestringWhich collection to save to. Defaults to "All Saved".
Save: POST /saved/rec_p9q8r7
Unsave: DELETE /saved/rec_p9q8r7
Example success response (200 OK)
{
  "success": true,
  "action": "saved",
  "collection_name": "Weekend Dinners",
  "recipe_id": "rec_p9q8r7",
  "saved_at": "2026-03-17T15:00:00Z"
}
💡
The same recipe can be saved to multiple collections. For example Ji-Yeon saves Marco's carbonara to both "Italian Classics" and "Quick Dinners". Each save is a separate row in saved_recipes with a different collection_name.
Part 5 — Search & media
GET /search
Search recipes and creators
The main search endpoint. Searches recipe titles, ingredients, tags, and creator usernames. Supports filters to narrow results. Returns both recipe and creator results in the same response.
🌐 Public
Query parameters
qstringThe search term. E.g. "chicken", "pasta", "@marco".
cuisinestringFilter by cuisine. E.g. "italian".
difficultystring"easy", "medium", or "hard".
max_timeintegerMax cook time in minutes.
min_ratingnumberMin average rating (1–5).
sortstring"relevant" (default), "newest", "top_rated", "most_saved".
pageintegerPage number for pagination.
Example:
GET /search?q=chicken&cuisine=korean&max_time=30&sort=top_rated
Example success response (200 OK)
{
  "query": "chicken",
  "total_results": 47,
  "recipes": [
    {
      "id": "rec_abc123",
      "title": "Korean BBQ Chicken Bowl",
      "cover_image_url": "https://cdn.rv.com/r/abc.jpg",
      "cook_time_mins": 25,
      "avg_rating": 4.9,
      "author": { "username": "jiyeon_eats" }
    }
  ],
  "creators": [
    { "username": "chickenlover99", "recipe_count": 24 }
  ],
  "has_more": true
}
💡
PostgreSQL has built-in full-text search that handles this without needing a separate search service. At MVP scale (under 100k recipes), this is fast enough. At larger scale, swap to Algolia or Typesense for better speed and typo tolerance.
POST /media/upload
Upload a photo or video
Uploads an image or video file to cloud storage (Uploadthing/Cloudinary) and returns the URL. This URL is then used when creating or editing a recipe. Always upload media first, then attach the URL to the recipe.
🔐 Auth required
Inputs (multipart/form-data)
fileFileThe image (JPG/PNG, max 10MB) or video (MP4, max 100MB).
typestring"image" or "video".
rolestring"cover", "step", "avatar", or "gallery".
⚠️ This endpoint uses multipart/form-data not JSON. The file is sent as binary data, not a text string.
Example success response (201 Created)
{
  "success": true,
  "media": {
    "id": "med_q1w2e3",
    "url": "https://cdn.rv.com/r/q1w2e3.jpg",
    "type": "image",
    "role": "cover",
    "width": 1200,
    "height": 800,
    "size_bytes": 245000
  }
}
💡
The upload flow is always: (1) Upload the file here → get back a URL. (2) Use that URL in the create/edit recipe call. Files are never stored in the database — only the URL string. The actual file lives in cloud storage which is built for serving images fast globally.
Standard error responses
Every API returns the same error format: { "success": false, "error": { "code": 401, "message": "You must be logged in to do this." } } — consistent errors make it easy for the frontend to handle them.
Status codeMeaningWhen it happensExample message
200 OKSuccessRequest worked, data returned"success": true
201 CreatedCreatedNew resource was successfully createdRecipe, comment, or user created
400 Bad RequestInvalid inputRequired field missing or wrong format"Title is required"
401 UnauthorizedNot logged inToken missing or expired"You must be logged in"
403 ForbiddenNo permissionLogged in but not allowed (e.g. editing someone else's recipe)"You don't own this recipe"
404 Not FoundDoesn't existRecipe or user ID doesn't exist in the database"Recipe not found"
409 ConflictDuplicateUsername taken, already liked, already following"Username already taken"
422 UnprocessableValidation failedData format correct but fails business rules"Rating must be between 1 and 5"
500 Server ErrorOur bugSomething unexpected broke on the server"Something went wrong. Try again."
The rule of good APIs: Be predictable. Every endpoint should behave the same way — same error format, same response structure, same auth pattern. When the frontend team can predict how any API works, they build faster and make fewer bugs.
