Before you read: Think of a database like a giant Excel spreadsheet. Each table is one spreadsheet. Each row is one record (one user, one recipe, one comment). Each column is one piece of information about that record. Tables are connected to each other using IDs — like how a recipe row stores the ID of the user who created it.

Part 1 — Identity & accounts

👤
users
Stores the core login information for every person who signs up. This is the most fundamental table — everything else connects back to it. Think of it as the master list of everyone on the platform.
Why separate from profiles? Users holds private auth data (email, password hash). Profiles holds public-facing info (bio, avatar). Keeping them separate is safer and cleaner.
→ profiles (1-to-1) → recipes (1-to-many) → followers (1-to-many) → likes (1-to-many) → comments (1-to-many)

Part 2 — Recipes & content

🍳
recipes
The heart of the entire platform. Every recipe anyone publishes lives here. This table stores the core info about a recipe — but not the ingredients or steps (those have their own tables to stay organised and flexible).
Ingredients and steps are in separate tables because a recipe can have 2 ingredients or 20. Storing them in a separate table lets us handle any number cleanly.
← users (many-to-1) → ingredients (1-to-many) → recipe_steps (1-to-many) → recipe_media (1-to-many) → likes (1-to-many) → comments (1-to-many) → saved_recipes (1-to-many)

Every ingredient for every recipe lives here. Each row is one ingredient in one recipe. A recipe with 8 ingredients = 8 rows in this table, all pointing to the same recipe.
Why not store ingredients as one big text field inside recipes? Because storing them separately lets users search by ingredient ("show me all recipes with chicken"), and lets us build features like shopping lists later.

Part 3 — Social interactions

👥
followers
Records every follow relationship between two users. When Ji-Yeon follows Marco, one row is added here. When she unfollows him, that row is deleted. Simple and elegant.
The combination of follower_id + following_id must always be unique — you can't follow the same person twice. A user also can't follow themselves (enforced in code).
← users as follower (many-to-1) ← users as following (many-to-1)
ColumnTypeDescriptionFlags
follower_idUUIDThe user who is doing the following. In "Ji-Yeon follows Marco" — this is Ji-Yeon's ID.
FK → usersRequiredIndexed
following_idUUIDThe user being followed. In "Ji-Yeon follows Marco" — this is Marco's ID.
FK → usersRequiredIndexed
created_atTIMESTAMPWhen the follow happened. Together, follower_id + following_id form the Primary Key — no duplicates allowed.
PK (composite)Required
❤️
likes
Records every time a user likes a recipe. One row = one like. When a user taps the heart on a recipe, a row is added. When they un-like it, the row is removed.
A user can only like the same recipe once — enforced by making user_id + recipe_id a unique combination.
← users (many-to-1) ← recipes (many-to-1)
ColumnTypeDescriptionFlags
user_idUUIDThe user who liked the recipe.
FK → usersPK (composite)Required
recipe_idUUIDThe recipe that was liked.
FK → recipesPK (composite)Required
created_atTIMESTAMPWhen the like happened.

💬
comments
Every comment on every recipe lives here. This table also supports replies — a comment can have a parent_comment_id pointing to another comment, creating a thread.
parent_comment_id is the clever trick for replies. If it's null, the comment is top-level. If it has a value, it's a reply to that parent comment. We support 2 levels deep (comment → reply).
← users (many-to-1) ← recipes (many-to-1) ← comments self-reference (reply to)

⭐
reviews
Stores star ratings and written reviews for recipes. Separate from comments because a review has a score (1–5 stars) and a user can only leave one review per recipe, while they can leave many comments.
Why separate from comments? Because reviews have a star rating and we enforce one-per-user. Comments are unlimited and have no score. Mixing them would make the data messy.

Records every recipe a user has saved to their collection. Each row is one saved recipe in one named collection. Example: Ji-Yeon saves Marco's carbonara to her "Italian Classics" collection.
collection_name lets users organise saves into folders like "Quick dinners", "Baking", "Dad's favourites". A user can save the same recipe to multiple collections.

Part 4 — Tags, categories & notifications

🏷️
tags
The master list of all tags on the platform. Examples: "vegan", "gluten-free", "30-minute meals", "comfort food", "street food". Each tag is created once and can be attached to many recipes.
Tags live in their own table so we never spell "vegetarian" differently across 1000 recipes. One tag row, many recipes use it.
→ recipe_tags (1-to-many)

RequiredUniqueIndexed
slugTEXTURL-friendly version. Example: "quick-meals". Used for tag pages like recipeverse.com/tag/quick-meals.
RequiredUnique
categoryTEXTGroups tags into types. Example: "dietary" (vegan, gluten-free), "meal-type" (breakfast, dinner), "technique" (baking, grilling).
Optional

Required
🔗
recipe_tags
This is called a "junction table" or "join table". It connects recipes and tags. One recipe can have many tags. One tag can be on many recipes. This table makes that many-to-many relationship possible.
Example: Marco's carbonara has tags "Italian", "pasta", "comfort food", "quick meals". That's 4 rows in this table, all with the same recipe_id but different tag_ids.
← recipes (many-to-1) ← tags (many-to-1)

notifications
Every in-app notification sent to every user. When Amara gets a new follower, a row is added here. When she opens the notification bell, all her unread notifications are fetched from this table.
The actor_id and entity fields make this table flexible. "actor liked your recipe" — actor_id is the liker, entity_type is "recipe", entity_id is the recipe ID. Works for any notification type.

Relationships between tables — plain English

How the tables connect to each other
users→profilesOne-to-one
Every user has exactly one profile. Created automatically at sign-up.
users→recipesOne-to-many
One user can publish many recipes. Every recipe belongs to one user.
recipes→ingredientsOne-to-many
One recipe has many ingredients. Each ingredient belongs to one recipe.
recipes→recipe_stepsOne-to-many
One recipe has many steps. Each step belongs to one recipe.
recipes→recipe_mediaOne-to-many
One recipe can have many photos and videos.
users↔usersMany-to-many (via followers)
Users follow other users. The followers table holds every follow relationship.
users↔recipesMany-to-many (via likes)
Many users can like many recipes. The likes table records each like.
users→commentsOne-to-many
One user can write many comments across any number of recipes.
comments→commentsSelf-reference (replies)
A comment can be a reply to another comment via parent_comment_id.
users→reviewsOne-to-one per recipe
A user can only leave one review per recipe but can review many recipes.
users↔recipesMany-to-many (via saved_recipes)
Many users can save many recipes into named collections.
recipes↔tagsMany-to-many (via recipe_tags)
One recipe can have many tags. One tag can be on many recipes.
users→notificationsOne-to-many
Every action on the platform can generate a notification for the relevant user.

Scalability notes — what to add as you grow

Add database indexes early
Every column marked Indexed in this document needs an index created in PostgreSQL. Indexes are like a book's index — they make lookups 100x faster. Add them before you have 10,000+ rows, not after.
Cache counts in profile rows
follower_count, recipe_count, like_count — these are stored directly on the row so you never have to count millions of rows just to show a number. Recalculate them in a background job every 5 minutes.
Soft deletes everywhere
Notice deleted_at and is_deleted fields instead of actually removing rows. This protects you — you can recover accidentally deleted content, and it keeps reply threads intact even if a parent comment is removed.
UUIDs instead of numbers
All IDs are UUIDs (random unique strings) not sequential numbers (1, 2, 3…). This means no one can guess IDs, you can merge databases without conflicts, and it scales across multiple servers.
Full-text search index
PostgreSQL has built-in full-text search. Add a search index on recipes.title and ingredients.name to power the search bar without needing a separate search service at MVP stage.
Future tables to add
When you reach Phase 2, you'll add: transactions (tips/payments), subscriptions (creator memberships), collections (user-named recipe folders as their own table), and reports (flagged content queue).
The golden rule of databases: Store each piece of information in exactly one place. If you find yourself copying the same data into multiple tables, something is wrong with the design. One source of truth — that's what makes a database clean and scalable.
